## Part 1: Azureの基本概念☁️

### Azureの階層構造🏢

Azureは階層構造で管理されています：

```
Azure AD（認証の大元）
└── Tenant（テナント = 会社全体）
    └── Management Groups（管理グループ = 部署みたいなもの）
        └── Subscriptions（サブスクリプション = 予算枠）
            └── Resource Groups（リソースグループ = プロジェクト）
                └── Resources（リソース = 実際のサーバーとか）
```

#### 1. Tenant（テナント）🏢
会社全体。普通は1社1テナント。

#### 2. Management Groups（管理グループ）🗂️
部署みたいなもの。階層構造で管理できます：

```
本社
├── 開発本部
│   ├── WebアプリチームSubscription
│   └── モバイルアプリチームSubscription
└── インフラ本部
    ├── ネットワークSubscription
    └── セキュリティSubscription
```

管理グループにポリシーを適用すると、配下の全サブスクリプションに継承されます。

#### 3. Subscriptions（サブスクリプション）💳
予算枠。月末に請求が来ます。  
プロジェクトや環境ごとに分けるのが一般的：

- 本番環境用Subscription
- 開発環境用Subscription
- 検証環境用Subscription

#### 4. Resource Groups（リソースグループ）📦
リソースを束ねる入れ物。  
「このプロジェクトで使うリソースは全部ここ」みたいに使います。

特徴：

- 同じリージョンでなくてもOK（リソースグループのリージョンと中のリソースのリージョンは揃えてね）
- まとめて削除できる
- まとめて権限管理できる

#### 5. Resources（リソース）🖥️
実際のVM、VNet、Storageなど。

---

### Azureの重要な概念💡

#### Location（リージョン）📍
Azureのデータセンターの場所。日本だと：

- `japaneast`（東日本 = 埼玉・東京）
- `japanwest`（西日本 = 大阪）

リージョンによって：

- 使えるサービスが違う
- 価格が違う
- 物理的に離れているので災害対策になる

#### Availability Zones（可用性ゾーン）🏢
1つのリージョン内の複数のデータセンター。  
例: 東日本リージョン内に3つのゾーン（Zone 1, 2, 3）

**重要**: 日本リージョンは全てのサービスでAvailability Zonesをサポートしてません！  
このプロジェクトでハマったポイントです。

#### Resource Provider🔌
Azureのサービス提供者。例：

- `Microsoft.Network`: VNet, Firewallなど
- `Microsoft.Compute`: VMなど
- `Microsoft.Storage`: ストレージ

---

### Azure Policy📜

Azureの「会社のルール」を強制する仕組み。

例：
```json
{
  "if": {
    "field": "type",
    "equals": "Microsoft.Network/publicIPAddresses"
  },
  "then": {
    "effect": "deny"
  }
}
```

「Public IPは作っちゃダメ」というルール。  
違反しようとすると、Azureが止めてくれます。

#### Policyの3要素🔑

1. **Policy Definition**（ポリシー定義）  
   ルールそのもの。「こうしなさい」「これダメ」

2. **Policy Set Definition**（ポリシーセット定義）  
   複数のポリシーをまとめたもの。「セキュリティ基準セット」とか。イニシアティブともいう。

3. **Policy Assignment**（ポリシー割り当て）  
   管理グループやサブスクリプションに適用する。

#### Policyの効果（Effect）🎯

- `Deny`: 拒否。作れない。
- `Audit`: 警告。作れるけど記録される。
- `Append`: 追加。タグを自動で付けるとか。
- `Modify`: 修正。設定を自動で変更。
- `DeployIfNotExists`: 無ければ作る。

## Part 2: Azure Landing Zonesの設計思想🛬

### Landing Zonesって何？

Microsoftが定義した「企業向けAzure環境のベストプラクティス」です。

これに沿って作ると、セキュリティ・ガバナンスが強い環境が勝手にできちゃいます！

---

### Landing Zonesの8つの設計領域🧩

Microsoftは8つの設計領域を定義しています。  
ランディングゾーンは、**これら全てを実装しています**。

---

#### 1. Azure Billing and Active Directory Tenant（課金とテナント）💰

**何を決めるか？**

- 誰がお金を払うか
- どうやって認証するか
- 契約形態はどうするか

**具体的には：**

![課金とテナント構造](./img/diagrams/lz-01-billing-tenant.svg)


**ベストプラクティス(抜粋)**

- 組織にActive Directoryがない場合は、EntraIDのみのIDデプロイを実装する。

  👉 EntraIDは新しいクラウドベースのID基盤であり、EntraID+Intuneを使ってADでやっていたことがクラウドのみでできます。

- 同じ組織に対して複数の Microsoft Entra テナントを作成しないでください。

  👉 この理由は後からテナントをまとめるのがすごく大変というのと、entra ID P2のライセンスが二重でかかる、DefenderやAzureポリシーのガバナンスが分かれるなどです。すでにM365を利用している会社でも、同テナントにAzure環境を構築するのが良いです。
  
- すべての Microsoft Entra 診断ログを中央の Azure Monitor Log Analytics ワークスペースに送信します

  👉 Entraのログはインシデント時にも重要な分析対象になるので、必ず中央のログストレージに保管します。

---

#### 2. Identity and Access Management（ID・アクセス管理）🔐

**何を決めるか？**

- 誰が何をできるか（RBAC）
- どうやってログインするか（MFA、Conditional Access）
- サービスアカウントの管理

**具体的には：**

![ID・アクセス管理](./img/diagrams/lz-02-identity-access.svg)


**ベストプラクティス(抜粋)**

- PIMや条件付きアクセス、MFA必須、EntraID Protectionを使う

  👉 IDのセキュリティは昨今のゼロトラストの流れの中で、最重要ポイントです。これらは必ず設定しましょう。

- 定期的にアクセスレビューで権限を棚卸し

  👉 最小特権の原則を実現するために定期的に権限を棚卸しましょう。

- Azureリソースに対する認証はマネージドIDや、Entra認証を使う

  👉 マネージドIDをつかると、資格情報のローテーションなどの管理の必要が無く、管理ミスがなくなります。また、Entra認証によるリソースの接続は、条件付きアクセスや、RBACの権限設定の恩恵が受けられるので強力な防御を実現できます。

---

#### 3. Resource Organization（リソースの整理）🗂️

**何を決めるか？**

- 管理グループの階層構造
- サブスクリプションの分け方
- リソースグループの命名規則

**具体的には：**

![リソースの整理](./img/diagrams/lz-03-resource-organization.svg)

**なぜこう分けるか？**

**1. Platform（基盤）は分離**

- 「インフラチーム」が管理
- アプリ開発者は触らせない
- 全社で共有（Firewall、監視等）

**2. Landing Zones（アプリ用）はさらに分ける**

- **Corp**: 社内専用（VPN経由のみ）
- **Online**: インターネット公開
- **Sandbox**: 実験用（コスト制限、自動削除）

**ベストプラクティス(抜粋)**

- 管理グループ階層は、3 ~ 4 レベル以下

  👉 管理の複雑さを軽減するため。ポリシーの割り当てや継承が分からなくなり、大きなミスに繋がります。

- アプリケーションチームには、管理グループスコープでRBACアクセス許可を割り当てない。

  👉 管理グループを管理できるのは、インフラチームだけにしましょう。セキュリティリスクが大幅に上がります。

- 命名規則を作る。

  👉 意外ですが、管理上重要です。

- ランディングゾーン（ユーザー用）は、Corp・Online・Sandboxの3つの管理グループに分ける。

  👉 ほとんどの企業はこの3つの管理グループの分け方で事足りるらしいです。

- 運用環境、テスト環境、開発環境用の管理グループは作成しないでください。 必要に応じて、これらのグループを同じ管理グループ内の異なるサブスクリプションに分割します。

  👉 管理グループの階層が深層化し管理が複雑になると記載があります。また、開発環境も本番と同等のガバナンスが必要になることが多いため、ポリシーを分ける効果があまり無い結果になる可能性があります。

- TerraformでデプロイしたリソースはTerraformで管理されていることが分かるタグをつける。

  👉 これは私が追加しました😊 後のTerraform以外でのリソースデプロイの制限に使います。



**命名規則の例：**

```
リソースグループ:
rg-{用途}-{環境}-{リージョン}
例: rg-web-prod-japaneast

リソース:
{種類}-{用途}-{環境}-{リージョン}
例: vm-web-prod-japaneast-001
```

**このプロジェクトでは：**

- Terraformで自動命名
- テンプレート変数で一括管理
- 全リソースに必須タグ（environment, cost-center等）

#### 4. Network Topology and Connectivity（ネットワーク）🌐

**何を決めるか？**

- ネットワーク構成（Hub-Spoke or Virtual WAN）
- オンプレミスとの接続方法
- IPアドレス設計

**具体的には：**

![ネットワーク構成](./img/diagrams/lz-04-network-topology.svg)

**オンプレミス接続の選択肢：**

| 接続方法 | 速度 | コスト | 特徴 |
|---------|------|--------|------|
| VPN Gateway | 最大1.25Gbps | ¥40,000/月 | インターネット経由、導入簡単 |
| ExpressRoute | 10Gbpsレベル | ¥50,000/月〜 | 専用線、安定、高額 |

**ベストプラクティス**

- HUBをリージョン冗長で配置する。それに伴って、AzureFirewall、ER回線とER Gatewayもリージョン冗長で配置することになる。

  👉 コストはかかるが、リージョン全体が停止したときの安心感があります。ここはSLAと相談です。

- アウトバウンドやER向けの通信はAzureFirewallを通す

  👉 AzureFirewallでの分析が行えるので、セキュリティが向上します。

- インバウンド通信はHubを通さない。(各Spokeにインバウンドする)

  👉 これは日本マイクロソフトのCAFにありました。Hubからインバウンド通信を隔離することで、Hubやその他接続環境への影響範囲を抑えることができます。

- すべてのネットワークで、VNETフローログを有効化

  👉 いつでも調査が行えるようにログを取っておく。AzureFirewallを通らないネットワークログもこれで取れるので、万が一の抜け道があったときの調査に役立ちます。

- DNSはHUBに集約し、プライベートエンドポイントのレコード登録を自動化する。

  👉 実装は少々難しいですが、DNS登録の手間がなくなります。

- Azure Virtual Network Managerを使用して、NSGの禁止ルールを定義する。

  👉 22,3389ポートへの受信を禁止する等、社内の絶対禁止ルールを定義できます。

- OnlineランディングゾーンのSpokeは会社ネットワークに接続できないよう制限し、利用者が自由にネットワークを設計できるようにする。

  👉 Online管理グループ配下のサブスクリプションは会社ネットワークから隔離されているため、利用者が自由にネットワークを設計できます。ゼロトラストアプリケーションの開発において利便性が向上します。

- CorpランディングゾーンのSpokeは会社ネットワーク(HUB)に接続するため、ネットワーク設定を制限する。

  👉 Corp管理グループ配下のサブスクリプションは会社ネットワークに接続するため、全社のネットワーク管理に従います。(IPAMなど)

- IPAMツールを使って、IPアドレス管理を自動化する。

  👉 IPAMによる自動化により、手動によるミスを防ぎます。

#### 5. Security（セキュリティ）🛡️

**何を決めるか？**

- Firewall/WAFの設定
- DDoS対策
- データ暗号化
- 脅威検知

**具体的には：**

![セキュリティ多層防御](./img/diagrams/lz-05-security-defense.svg)


**ベストプラクティス：**

- すべてのサブスクリプションでDefender for Cloudを有効化する。Defender for Cloudのアラートを有効化する。

  👉 セキュリティ情報の収集やインシデント対応ができます。

- ゼロトラストを意識して設計する。

  👉 ゼロトラストはモダンなセキュリティ思想であり、セキュリティ強度と利便性を兼ね備えています。一般的にはセキュリティを強化すると利用者の利便性が下がるのですが、ゼロトラストはそうじゃないのがいいと思います。

  ※ゼロトラストはAzureもそうですが、主にEntraID・Microsoft Intune・Defender XDRの設計が重要です。

#### 6. Management（管理）🛠️

**何を決めるか？**

- 監視・ログ収集
- バックアップ戦略
- 災害対策（DR）

**具体的には：**

![管理とモニタリング](./img/diagrams/lz-06-management-monitoring.svg)


**ベストプラクティス**

- ログは単一のLog Analyticsワークスペースにすべて集約する。

  👉 この理由はSOCやインシデント対応の際の横断分析のため。Logの保存先が分かれていると、検索結果の表結合による分析の難易度が跳ね上がる可能性があります。また、コミット料金による料金削減もできます。

- GitHubでインフラ更新のすべての履歴が残るように管理

  👉 後から、だれが変更したかどう変更したかが残るので基盤管理者としては安心です。GitHubで管理しておけば、もしランサムウェアでインフラを破壊されても復元できます。

#### 8. Platform Automation and DevOps（自動化）🤖

**何を決めるか？**

- CI/CDパイプライン
- Infrastructure as Code（IaC）
- 自動テスト

**具体的には：**

![DevOps自動化](./img/diagrams/lz-08-devops-automation.svg)

**このプロジェクトでは：**

- GitHub Actionsでパイプライン構築
- Terraform Cloudでステート管理
- Azureへの認証はOIDC（パスワード不要）

**自動化のメリット：**

- 人的ミスが減る
- デプロイが早い（数分）
- 履歴が残る（いつ誰が何を変更したか）
- ロールバック簡単（Gitで戻すだけ）

**ベストプラクティス**

- GitHubを使用して、すべての更新履歴を追跡できるように管理する。

  👉 更新履歴が全部残るので安心。

- インフラの更新はコード更新→承認→自動テスト→自動デプロイというフローにすべて従う。

  👉 レビューしていない変更がなくなるので安心。勝手にインフラを変更できなくなります。

- 運用作業もコードで定義し、自動化する（コードがパラメーターファイルの代わりになる。）

  👉 手動作業によるミスや、レビュー漏れがなくなる。パラメーターとして残るので、あとから参加したメンバーもここを見れば基盤を理解できます。

- mainブランチを保護し、承認無しでインフラが更新されないようにする。

  👉 mainブランチの更新＝CDによる本番インフラの更新です。mainブランチを保護することで、必ず変更前にレビューを通せます。

- AzureポータルやAzure CLIでの設定変更をしない。GitHub Actionsで手動更新を定期的に検知する。

  👉 パラメーターファイルとしてもIaCを使うなら、すべてをTerraformで管理するべきですよね。

## Part 3: Infrastructure as Code (IaC)💻

### なぜコードで書くのか？

#### 従来の問題点⚡
```text title="手順書の問題"
手順書: サーバー構築手順.docx

1. ポータルにログイン
2. リソースグループ「rg-prod」を作成
3. 場所は「East Japan」を選択
4. ...（省略）...
20. 完了
```

これの問題：

- 人によって解釈が違う
- 手順書が古くなる（特にクラウドはすぐ変わる）
- ミスが起きる
- 再現が大変

#### IaCのメリット🌟

**1. 再現性**
```hcl title="再現性の例"
resource "azurerm_resource_group" "prod" {
  name     = "rg-prod"
  location = "japaneast"
}
```

何度実行しても同じ結果。

**2. バージョン管理**
```bash title="Git履歴"
git log
# commit 3: Firewallを追加
# commit 2: VNetのサイズを変更
# commit 1: 初期構築
```

いつ誰が何を変更したかわかる。

**3. レビュー**
```bash title="差分確認"
git diff
# + resource "azurerm_firewall" "fw" {
# +   name = "fw-prod"
# + }
```

変更内容を誰かにチェックしてもらえる。

**4. 自動化**
```yaml title="GitHub Actions"
# GitHub Actions
on: push
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - terraform apply
```

GitにpushしたらFull自動でデプロイ。

**5. ドキュメント**

コード自体がドキュメント。  
「どう作ったか」が明確。
パラメーターシートを作らなくていい。
Document as Code = 設計書などのDocumentもGitHubで管理するのはおすすめ。
※Microsoftの公式ドキュメントはGitHubで管理されてます。

---

### IaCのベストプラクティス🏆

#### 1. 小さく始める🔰
いきなり全部IaCにしない。  
まずは1つのリソースから。

#### 2. モジュール化🧱
再利用可能な部品に分ける。  
このプロジェクトはモジュール化の良い例。

#### 3. 環境を分ける🌏
```text title="環境の分離"
environments/
├── dev/      # 開発環境
├── staging/  # 検証環境
└── prod/     # 本番環境
```

#### 4. シークレットを含めない🔑
パスワードやAPIキーはコードに書かない。絶対にやめましょう。
環境変数(GitHub Secret)やKey Vaultを使いましょう。

#### 5. Planを必ず見る👀
`terraform apply`の前に`terraform plan`。  
何が変わるか必ず確認。

## Part 4: Terraformって何？🌱

### 「コードでインフラを作る」とは？

=== "昔のやり方 🏢"

    ```text title="オンプレミス時代"
    1. データセンターに行く
    2. ラックからサーバーを取り出す
    3. ケーブルを繋ぐ
    4. OSをインストール
    5. 設定ファイルを手で編集
    ```

=== "今のやり方（クラウド） ☁️"

    ```text title="Azureポータルでの作成"
    1. Azureポータルにログイン
    2. 「仮想マシン」をクリック
    3. フォームに入力（名前、サイズ、場所...）
    4. 「作成」ボタンをクリック
    ```

!!! warning "でも、これって問題がある ⚠️"
    - 間違えやすい（入力ミス）
    - 再現できない（どう作ったか忘れる）
    - 時間がかかる（10個作るなら10回やる）

そこでTerraform。コードでインフラを定義します：

```hcl title="Terraformでの定義"
resource "azurerm_virtual_machine" "example" {
  name                = "my-vm"
  location            = "japaneast"
  size                = "Standard_B2s"
  # ...その他の設定
}
```

このコードを実行すると、VMが作られます。

!!! success "コードで管理するメリット✨"
    - バージョン管理できる（Git）
    - レビューできる
    - 再利用できる
    - 自動化できる

---

### Terraformとは❓

Terraformとは、HashiCorp社が開発したインフラをコードで管理するIaCの仕組みであり、

特徴的なのは、AWS,Azure,GCPなど様々なサービスを同じ言語で記述できることです。

マルチクラウドでの利用、シェアが高いので参考情報が多いなどのメリットがあります。

---

### Terraformの3つのステップ🪜

Terraformは3ステップで動きます：

![Terraformワークフロー](./img/diagrams/terraform-workflow.svg)

#### 1. Write（書く）✍️
```hcl title="main.tf"
resource "azurerm_resource_group" "example" {
  name     = "my-rg"
  location = "japaneast"
}
```

#### 2. Plan（計画）📝
```bash title="実行前の確認"
$ terraform plan
# Terraform will perform the following actions:
#   + azurerm_resource_group.example
#       name:     "my-rg"
#       location: "japaneast"
```

「これから何を作るか」を教えてくれます。

#### 3. Apply（適用）🚀
```bash title="実際に作成"
$ terraform apply
# 本当に実行してもいい？ (yes/no): yes
# Creating...
# Creation complete!
```

実際にAzureにリソースが作られます。

---

### Terraformの賢いところ🧠

#### 差分管理🔄
Terraformは「現在の状態」を覚えています。

```hcl title="設定の変更例"
# 最初はこう書いた
resource "azurerm_resource_group" "example" {
  name     = "my-rg"
  location = "japaneast"
}

# 後で場所を変更
resource "azurerm_resource_group" "example" {
  name     = "my-rg"
  location = "japanwest"  # ← 変更
}
```

`terraform apply`すると：

- リソースグループを削除して
- 新しい場所に作り直す

Terraformが賢く判断してくれます。

---

#### 依存関係の解決🔗
```hcl title="依存関係の自動解決"
resource "azurerm_resource_group" "rg" {
  name     = "my-rg"
  location = "japaneast"
}

resource "azurerm_virtual_network" "vnet" {
  name                = "my-vnet"
  resource_group_name = azurerm_resource_group.rg.name  # ← 依存
  location            = azurerm_resource_group.rg.location
  address_space       = ["10.0.0.0/16"]
}
```

VNetを作る前に、RGを作る必要があります。  
Terraformが自動で順序を判断してくれます。

### HCLの基本文法📑

TerraformはHCL（HashiCorp Configuration Language）で書きます。

HCLとは、HashiCorp 社が開発したIacに特化した言語です。この言語は、人間にとって読みやすく、JSON と互換性があるのが特徴です。

組み込みの関数もあり、変数操作などのちょっと込み入った処理もこなせます。

#### ブロック構造🧩

terraformのリソース定義は以下の形式が基本です。

リソース定義は「main.tf」というファイルに書くことが多いです。

※「main.vnet.tf」みたいにリソース種類ごとに分けることもあります。

```hcl title="HCLの基本構造"
<ブロックタイプ> "<ラベル>" "<名前>" {
  <引数> = <値>
}
```

例：
```hcl
resource "azurerm_resource_group" "example" {
  name     = "my-rg"
  location = "japaneast"
}
```

- `resource`: リソースブロックを定義。リソースを作成することを宣言します。
- `"azurerm_resource_group"`: リソースのタイプ。リソースグループとかVNetとかいろいろ。
- `"example"`: Terraform内での識別名。これはTerraformの中だけの名前。
- `name`, `location`: 引数。ここでリソース名とか、リージョン、設定を入れます。

#### 変数定義🔢

terraformでは変数を使えます。型やデフォルト値の指定、バリデーション（値のチェック）までできます。

変数定義は「variables.tf」というファイルにまとめることが多いです。

```hcl
# 変数定義
variable "location" {
  default = "japaneast"
}
```

#### 変数参照🔍

変数の参照方法は以下の感じです。var.xxxみたいにして変数を参照します。

```hcl
# 変数使用
resource "azurerm_resource_group" "rg" {
  name     = "my-rg"
  location = var.location  # ← var.で参照
}
```

#### 変数の渡し方🔍

変数を渡すには、terraform.tfvarsまたはterraform.auto.tfvarsなどの名前のファイルに定義して渡します。

```hcl
# 変数定義
location = "japaneast"
```

#### 他のリソース参照🔗

リソースブロックで定義したTerraformの中での識別名でその中身を参照できます。

サブネットにどのルートテーブルを紐づけるか指定するためにIDを参照するみたいに使います。

```hcl
resource "azurerm_resource_group" "rg" {
  name     = "my-rg"
  location = "japaneast"
}

resource "azurerm_virtual_network" "vnet" {
  name                = "my-vnet"
  resource_group_name = azurerm_resource_group.rg.name  # ← 他のリソースを参照
  location            = azurerm_resource_group.rg.location
}
```

#### よくあるファイル構成

良くあるファイル構成を紹介します。
locals.tfはないこともありますが、それ以外はほぼほぼこの形がテンプレです。

![Terraformファイル構成](img/diagrams/terraform-file-structure.svg)

それぞれのファイルの役割を理解すると、Terraformのコードが読みやすくなります 📖

- **terraform.tf**：Terraform自体の設定（バージョン、プロバイダー、バックエンド）
- **variables.tf**：変数を定義（箱を用意するイメージ）
- **terraform.tfvars**：変数に実際の値を入れる
- **locals.tf**：変数を組み合わせて新しい値を作る（下ごしらえ）
- **main.tf**：リソースの本体を定義（メイン処理）
- **outputs.tf**：実行結果を出力

## 練習問題📝

練習問題で理解度チェックだ＾＾

### 問題1
Terraformの`terraform plan`コマンドは何をするコマンドですか？

### 問題2
Azureで「管理グループ」の役割は何ですか？

### 問題3
IaCの最大のメリットを1つ挙げてください。

### 問題4
このプロジェクトで使っているterraformのプロバイダーを3つ挙げてください。

---

## 練習問題の答え💡

### 答え1
`terraform plan`は、Terraformが実際にリソースを変更する前に、「何が作成/変更/削除されるか」を表示するコマンドです。実際には何も変更せず、プレビューだけを見せてくれます。

### 答え2
管理グループは、複数のサブスクリプションをまとめて管理するための階層構造です。ポリシーやアクセス制御を一括で適用でき、組織全体のガバナンスを効率的に実現できます。

### 答え3
IaCの最大のメリットは「再現性」です。同じコードから何度でも同じ環境を作れるため、手作業のミスがなくなり、環境の差異も発生しません。ランサムウェア攻撃への最大の対応策にもなります。

### 答え4
このプロジェクトで使っているterraformのプロバイダーは：

1. **alz** (Azure Landing Zones専用プロバイダー)
2. **azurerm** (Azureリソース管理)
3. **azapi** (最新Azure API対応)
