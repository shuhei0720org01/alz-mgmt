## 📖 このChapterを読む前に知っておくべきこと

### 🧩 テンプレート変数について

このファイルでは **テンプレート変数** を多用しています：

```hcl title="テンプレート変数の例"
management_resource_group_name = "rg-management-$${starter_location_01}"
```

**🔎 何が起きるか**:

- `$${starter_location_01}` → `japaneast` に自動置換される
- 実際の名前: `"rg-management-japaneast"`

**🤔 Terraformの変数（`${}`）との違い**:

| 記法 | 処理タイミング | 用途 |
|------|--------------|------|
| `${}` | Terraform実行時（即座） | Terraform変数の参照 |
| `$${}`  | テンプレート処理時（後で） | 設定ファイル内の置換 |

**📝 よく使われる置換変数**:

- `$${starter_location_01}` → 最初のリージョン（例: `japaneast`）
- `$${starter_location_02}` → 2番目のリージョン（例: `japanwest`）
- `$${starter_location_01_short}` → 短縮名（例: `jp`）

詳細は [Chapter 09: 設定テンプレート](./09_設定テンプレート.md) で解説します。
まずは「自動で置き換わる」と理解すればOKです。

## 🗾 Part 1: リージョン設定

### 🌍 starter_locations

```hcl title="リージョン設定"
starter_locations = ["japaneast", "japanwest"]
```

**❓ これは何でしょうか？**

- このプロジェクトで使うAzureリージョンのリストです
- 最初の項目（`japaneast`）がプライマリ（メイン）
- 2番目の項目（`japanwest`）がセカンダリ（サブ）

**🔄 どう使われるのでしょうか？**
後で出てくる設定で、以下のように自動置換されます：

```hcl title="自動置換の仕組み"
# 設定ファイル内で
resource_group_name = "rg-hub-$${starter_location_01}"

# 実際には
resource_group_name = "rg-hub-japaneast"  # ←自動で置き換わる
```

**🛠️ 変えたい時は？**
```hcl title="リージョン設定の変更例"
# 東京だけにしたい
starter_locations = ["japaneast"]

# アメリカにしたい
starter_locations = ["eastus", "westus"]

# 3リージョン使いたい
starter_locations = ["japaneast", "japanwest", "eastasia"]
```

簡単ですよね。

## 🧩 Part 2: カスタム置換 - names セクション

ここからが本番です。`custom_replacements`の中身を見ていきます。

### ✉️ 2-1. Defenderのメール設定

```hcl title="セキュリティアラート通知先"
custom_replacements = {
  names = {
    defender_email_security_contact = "security@example.com"
```

**❓ これは何でしょうか？**

- Microsoft Defender for Cloud（セキュリティ監視サービス）が脅威を検出したら、ここに通知が飛んできます
- セキュリティアラートのメールアドレスです

**🛠️ 変更する場合：**
```hcl title="メールアドレスの変更"
defender_email_security_contact = "security-team@yourcompany.com"
```

自分のメールアドレスに変更してください。

### 📦 2-2. リソースグループ名

```hcl title="リソースグループの命名"

    management_resource_group_name                 = "rg-management-$${starter_location_01}"
    connectivity_hub_primary_resource_group_name   = "rg-hub-$${starter_location_01}"
    connectivity_hub_secondary_resource_group_name = "rg-hub-$${starter_location_02}"
    dns_resource_group_name                        = "rg-hub-dns-$${starter_location_01}"
    ddos_resource_group_name                       = "rg-hub-ddos-$${starter_location_01}"
    asc_export_resource_group_name                 = "rg-asc-export-$${starter_location_01}"
```

**❓ これは何？**

Azureのリソースグループ（リソースを入れる箱）の名前を決めてる。

**🔎 実際の名前は？**

`$${starter_location_01}`が`japaneast`に置き換わるから：

| 変数名 | 実際の名前 | 何を入れる箱？ |
|--------|-----------|---------------|
| `management_resource_group_name` | `rg-management-japaneast` | Log Analyticsとか管理系 |
| `connectivity_hub_primary_resource_group_name` | `rg-hub-japaneast` | プライマリHubのネットワーク |
| `connectivity_hub_secondary_resource_group_name` | `rg-hub-japanwest` | セカンダリHubのネットワーク |
| `dns_resource_group_name` | `rg-hub-dns-japaneast` | Private DNS Zones |
| `ddos_resource_group_name` | `rg-hub-ddos-japaneast` | DDoS Protection |
| `asc_export_resource_group_name` | `rg-asc-export-japaneast` | Defender データエクスポート |

**📛 命名規則**

`rg-{用途}-{リージョン}`って感じで統一されてるでしょ？
これがAzureの推奨命名規則。

**🛠️ 変えたい時は？**
```hcl title="カスタム命名規則"
# 会社のルールに合わせたい
management_resource_group_name = "rg-prod-mgmt-$${starter_location_01}"
connectivity_hub_primary_resource_group_name = "rg-prod-network-$${starter_location_01}"
```

### 🛡️ 2-3. 管理リソースの名前

```hcl title="管理系リソースの命名"
    # Resource names management
    log_analytics_workspace_name            = "law-management-$${starter_location_01}"
    ddos_protection_plan_name               = "ddos-$${starter_location_01}"
    ama_user_assigned_managed_identity_name = "uami-management-ama-$${starter_location_01}"
    dcr_change_tracking_name                = "dcr-change-tracking"
    dcr_defender_sql_name                   = "dcr-defender-sql"
    dcr_vm_insights_name                    = "dcr-vm-insights"
```

**🔍 それぞれ解説します**

#### 📝 log_analytics_workspace_name
```hcl title="Log Analytics Workspaceの名前"
log_analytics_workspace_name = "law-management-$${starter_location_01}"
# 実際: "law-management-japaneast"
```

**❓ これは何でしょうか？**

- Log Analytics Workspace（ログを集めるデータベース）の名前です
- `law`はLog Analytics Workspaceの略
- すべてのログがここに集まります

#### 🛡️ ddos_protection_plan_name
```hcl title="DDoS Protectionの名前"
ddos_protection_plan_name = "ddos-$${starter_location_01}"
# 実際: "ddos-japaneast"
```

**❓ これは何でしょうか？**

- DDoS攻撃から守るサービスの名前です
- 非常に高額です（月¥400,000程度）
- インターネット公開しているアプリには必要です

#### 🆔 ama_user_assigned_managed_identity_name
```hcl title="Azure Monitor Agent用の認証ID"
ama_user_assigned_managed_identity_name = "uami-management-ama-$${starter_location_01}"
# 実際: "uami-management-ama-japaneast"
```

**❓ これは何でしょうか？**

- Azure Monitor Agent（監視エージェント）が使う認証IDです
- `uami`はUser Assigned Managed Identityの略
- VMなどにエージェント導入する際に使います

#### 📊 dcr_*_name（Data Collection Rule）
```hcl title="データ収集ルールの名前"
dcr_change_tracking_name = "dcr-change-tracking"
dcr_defender_sql_name    = "dcr-defender-sql"
dcr_vm_insights_name     = "dcr-vm-insights"
```

**❓ これは何？**

- Data Collection Rule（データ収集ルール）の名前
- どのデータを収集するか決めるルール
- 3種類：
  - `change-tracking`：ファイルとか設定の変更を追跡
  - `defender-sql`：SQL Serverのセキュリティログ
  - `vm-insights`：VMのパフォーマンスデータ

### 🔄 2-4. リソースの有効/無効設定（グローバル）

### 🏢 2-5. プライマリHubの設定

```hcl title="プライマリHubのリソース有効/無効設定"
    # Resource provisioning primary connectivity
    primary_firewall_enabled                                             = true
    primary_firewall_management_ip_enabled                               = true
    primary_virtual_network_gateway_express_route_enabled                = true
    primary_virtual_network_gateway_express_route_hobo_public_ip_enabled = true
    primary_virtual_network_gateway_vpn_enabled                          = true
    primary_private_dns_zones_enabled                                    = true
    primary_private_dns_auto_registration_zone_enabled                   = true
    primary_private_dns_resolver_enabled                                 = true
    primary_bastion_enabled                                              = true
```

**🌟 これが一番重要です！**
プライマリHub（Japan East）で何を有効にするか決めています。

#### 🔍 1つずつ見ていきましょう

##### 🔥 primary_firewall_enabled
```hcl
primary_firewall_enabled = true
```

**❓ これは何でしょうか？**

- Azure Firewall（ファイアウォール）を作るかどうかです
- `true`：作ります（月¥170,000程度、Standard SKU）
- `false`：作りません

**🛠️ いつfalseにしますか？**
```hcl title="コスト削減のためにFirewallを無効化"
# コスト削減したい検証環境
primary_firewall_enabled = false

# 代わりにNSG（無料のファイアウォール）だけ使う
```

##### 🌐 primary_firewall_management_ip_enabled
```hcl
primary_firewall_management_ip_enabled = true
```

**❓ これは何でしょうか？**

- Firewall管理用のPublic IPを作るかどうかです
- Basic SKUのFirewallで必要です
- Standard/Premiumなら不要ですが、作っておいても問題ありません

##### 🚇 primary_virtual_network_gateway_express_route_enabled
```hcl
primary_virtual_network_gateway_express_route_enabled = true
```

**❓ これは何でしょうか？**

- ExpressRoute Gateway（専用線ゲートウェイ）を作るかどうかです
- オンプレとAzureを専用線で繋ぐ時に必要です
- 月¥40,000〜60,000程度です（SKUによる）

**🛠️ いつfalseにしますか？**
```hcl title="オンプレ接続が不要な場合"
# オンプレとの接続が不要なら
primary_virtual_network_gateway_express_route_enabled = false
```

##### 🔑 primary_virtual_network_gateway_vpn_enabled
```hcl
primary_virtual_network_gateway_vpn_enabled = true
```

**❓ これは何？**

- VPN Gateway（VPNゲートウェイ）を作るか
- オンプレとAzureをインターネット経由で繋ぐ
- 月¥40,000程度（VpnGw1 SKU）

**🤔 ExpressRouteとの違いは？**
- ExpressRoute：専用線（速い、高い、安定）
- VPN：インターネット経由（普通、安い、不安定）

##### 🗂️ primary_private_dns_zones_enabled
```hcl
primary_private_dns_zones_enabled = true
```

**❓ これは何？**

- **Private DNS Zones**（Azure内部専用のDNSサービス）を作るか
- Azure PaaSサービス（Storage、SQL Database等）にプライベートエンドポイントで接続する時に必須
- 絶対`true`にしておこう

##### 📝 primary_private_dns_auto_registration_zone_enabled
```hcl
primary_private_dns_auto_registration_zone_enabled = true
```

**❓ これは何？**

- VMを作ったら自動でDNSに登録するか
- `true`にしておくとVMの名前で通信できて便利

##### 🧩 primary_private_dns_resolver_enabled
```hcl
primary_private_dns_resolver_enabled = true
```

**❓ これは何？**

- Private DNS Resolver（DNS中継サーバー）を作るか
- オンプレとAzure間でDNS名前解決する時に必要

##### 🛡️ primary_bastion_enabled
```hcl
primary_bastion_enabled = true
```

**❓ これは何？**

- **Azure Bastion**（安全なVMリモート接続サービス）を作るか
- VMにSSH/RDPする時、Bastionを経由すると安全
- Public IPをVMに割り当てなくて済む
- 月¥27,000程度（Basic SKU）

**💡 ポイント**

Japan EastではAvailability Zones使えないから、後で`zones = []`設定が必要になる。

## Part 3: 設定値のアドレス空間

### 3-1. IPアドレスの設計（プライマリ）

📚 IPアドレスとCIDR表記について

ここから`10.0.0.0/16`のような表記が出てきます。
これはCIDR表記（サイダー表記）と呼ばれるIPアドレスの範囲を表す記法です。

CIDR表記の読み方:
```
10.0.0.0/16
│      │
│      └─ ネットワーク部分のビット数（/16 = 最初の16ビット固定）
└─ 開始IPアドレス
```

**よく使うサイズ（覚えなくてOK、雰囲気だけ）**:

- `/16`: 65,536個のIPアドレス（大規模ネットワーク全体）
- `/22`: 1,024個（Hub VNet用）
- `/24`: 256個（一般的なサブネット）
- `/26`: 64個（Firewall等の専用サブネット）
- `/27`: 32個（Gateway用）
- `/28`: 16個（DNS Resolver用）

数字が小さいほど広い範囲です。詳しく理解する必要はありません。  
「大きな範囲を小さく区切って使う」と思えばOKです。

---

```hcl title="プライマリHubのIPアドレス設定"
    # IP Ranges Primary
    # Regional Address Space: 10.0.0.0/16
    primary_hub_address_space                          = "10.0.0.0/16"
    primary_hub_virtual_network_address_space          = "10.0.0.0/22"
    primary_firewall_subnet_address_prefix             = "10.0.0.0/26"
    primary_firewall_management_subnet_address_prefix  = "10.0.0.192/26"
    primary_bastion_subnet_address_prefix              = "10.0.0.64/26"
    primary_gateway_subnet_address_prefix              = "10.0.0.128/27"
    primary_private_dns_resolver_subnet_address_prefix = "10.0.0.160/28"
```

**📊 IPアドレスの全体像（視覚化）**

```
10.0.0.0/16（リージョン全体: 65,536個のIPアドレス）
├─ 10.0.0.0/22（Hub VNet: 1,024個）
│  ├─ 10.0.0.0/26    [Firewall Subnet]          64個
│  ├─ 10.0.0.64/26   [Bastion Subnet]           64個
│  ├─ 10.0.0.128/27  [Gateway Subnet]           32個
│  ├─ 10.0.0.160/28  [DNS Resolver Subnet]      16個
│  └─ 10.0.0.192/26  [Firewall Mgmt Subnet]     64個
│
└─ 10.0.4.0〜10.0.255.255（残り: 64,512個）
   └─ リージョン内のSpokeに予約
```

**表形式でも見てみましょう**

| 用途 | CIDR | IP範囲 | 使える数 |
|------|------|--------|----------|
| リージョン全体 | `10.0.0.0/16` | 10.0.0.0 - 10.0.255.255 | 65,536個 |
| Hub VNet | `10.0.0.0/22` | 10.0.0.0 - 10.0.3.255 | 1,024個 |
| Firewall | `10.0.0.0/26` | 10.0.0.0 - 10.0.0.63 | 64個 |
| Firewall Management | `10.0.0.192/26` | 10.0.0.192 - 10.0.0.255 | 64個 |
| Bastion | `10.0.0.64/26` | 10.0.0.64 - 10.0.0.127 | 64個 |
| Gateway | `10.0.0.128/27` | 10.0.0.128 - 10.0.0.159 | 32個 |
| DNS Resolver | `10.0.0.160/28` | 10.0.0.160 - 10.0.0.175 | 16個 |

**🎯 設計のポイント**

- Hub VNetは`10.0.0.0/22`（1,024個）だけ使う
- 残りの64,512個はリージョン内のSpoke用に予約
- 各Subnetが重ならないように慎重に設計されている
- Firewallが2つ（本体+管理用）あることに注目

### テレメトリー設定

```hcl title="enable_telemetry"
enable_telemetry = true
```

**テレメトリーとは？**

Microsoftにモジュールの利用状況（どのモジュールがどのくらい使われているか）を匿名で送信する機能です。

**何が送信される？**

- 使用しているモジュールのバージョン
- リージョン情報
- リソースの種類

**個人情報は送信される？**

いいえ。サブスクリプションID、リソース名、設定値などは一切送信されません。

**なぜデフォルトで有効？**

Microsoftがモジュールの改善に使うデータを収集するためです。  
オープンソースプロジェクトの維持に役立ちます。

**無効にしたい場合:**

```hcl
enable_telemetry = false
```

企業ポリシーで外部への通信が制限されている場合は、`false`にしてください。

#### 🏷️ Private DNS Zones設定

```hcl title="Private DNSゾーン設定"
    private_dns_zones = {
      parent_id = "$${dns_resource_group_id}"
      private_link_private_dns_zones_regex_filter = {
        enabled = false
      }
      auto_registration_zone_enabled = "$${primary_private_dns_auto_registration_zone_enabled}"
      auto_registration_zone_name    = "$${primary_auto_registration_zone_name}"
    }
```

**Private DNS Zonesとは？🔍**

Azure内部でのみ使える名前解決（DNS）の仕組みです。

**主な用途:**

1. **Private Link用のDNSゾーン**: Azure Storage、SQL Database等のPrivate Endpointの名前解決
2. **自動登録ゾーン**: VMが起動すると自動的にホスト名が登録される

**private_link_private_dns_zones_regex_filter.enabled**
```hcl
enabled = false
```

- `false`: すべてのPrivate Link用DNSゾーンを作成
- `true`: 正規表現でフィルタリングして一部のみ作成

通常は`false`（全部作成）でOKです。

**auto_registration_zone**
```hcl
auto_registration_zone_enabled = "$${primary_private_dns_auto_registration_zone_enabled}"
auto_registration_zone_name    = "$${primary_auto_registration_zone_name}"
```

- VMが起動すると、自動的に`japaneast.azure.local`に登録される
- 他のVMから`vm-app01.japaneast.azure.local`で名前解決できる

#### 🧩 Private DNS Resolver設定

```hcl title="Private DNS Resolver設定"
    private_dns_resolver = {
```

**Private DNS Resolverとは？🔍**

オンプレミスとAzure間でDNS解決を可能にするサービスです。

**使用例:**

- オンプレ → Azure: オンプレのサーバーがAzure内のVMを名前で解決
- Azure → オンプレ: Azure内のVMがオンプレのサーバーを名前で解決

**なぜ必要？🤔**

従来のVPN経由のDNS転送よりも、高速で安定したDNS解決ができます。

**コスト💰:**

- 約¥8,000/月 + クエリ課金

オンプレ接続がない環境では無効化してコスト削減できます。

1. **名前の定義**（`names`セクション）
2. **有効/無効の設定**（`xxx_enabled`）
3. **実際のリソース設定**（各セクション）

**覚えておくべきポイント📝**：

- リージョンは`starter_locations`で決まります
- 有効/無効は`xxx_enabled = true/false`で設定します
- コスト削減したいならFirewall、VPN、Bastionを`false`に設定します

---

## 🏋️‍♂️ 練習問題

理解度チェックです。休憩中に考えてみましょう。

### ❓ 問題1
このプロジェクトを西日本（japanwest）だけにデプロイするには、  
`starter_locations`をどう変更すればいいですか？

### ❓ 問題2
Firewallを無効化してコストを削減したい場合、  
どの設定を`false`に変更すればいいですか？（具体的な変数名を答えてください）

### ❓ 問題3
`management_resource_group_name = "rg-management-$${starter_location_01}"`という設定で、  
`starter_locations = ["japaneast", "japanwest"]`の場合、  
実際のリソースグループ名は何になりますか？

---

## 📝 練習問題の答え

### 💡 答え1
```hcl title="西日本だけにデプロイ"
starter_locations = ["japanwest"]
```
1つのリージョンだけ指定します。セカンダリは不要なので削除。

### 💡 答え2
```hcl title="Firewallを無効化してコスト削減"
primary_firewall_enabled = false
secondary_firewall_enabled = false
```
プライマリとセカンダリ両方のFirewallを無効化します。  
これで月額約28万円のコスト削減になります。

### 💡 答え3
```
rg-management-japaneast
```
`$${starter_location_01}`が最初のリージョン`japaneast`に置き換わります。
