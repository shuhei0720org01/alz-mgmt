    08. 管理グループとポリシー - ガバナンスの要¶

    8つの設計領域との対応¶

    このChapterは以下の設計領域を実装します： - ✅ 2. Identity and Access Management（ID・アクセス管理 - RBAC、権限管理） - ✅ 3. Resource Organization（リソース整理 - 管理グループ階層） - ✅ 5. Security（セキュリティ - セキュリティポリシー） - ✅ 7. Governance（ガバナンス - ポリシー定義と適用）

    → Chapter 01で学んだ8つの設計領域を復習したい方は01_基礎知識.mdへ

    このChapterでやること¶

    管理グループとポリシーの仕組みを理解しよう。

    管理グループって何？ サブスクリプションを束ねる「フォルダ」みたいなもん。

    ポリシーって何？ 「これやっちゃダメ」「これやらないとダメ」みたいなルール。

    例えるなら： - 管理グループ：会社の部署（営業部、開発部、総務部） - ポリシー：社内規定（「経費は1万円まで」「セキュリティ研修必須」） - サブスクリプション：各チーム

    管理グループの基本¶

    階層構造¶

    📊 このプロジェクトの管理グループ階層（完全版）

    Tenant Root Group（最上位）
│
└── alz（Azure Landing Zones）
    │
    ├── platform（プラットフォーム層）─ インフラ基盤チーム
    │   ├── management（管理）─ ログ・監視システム
    │   │   └── [Subscription: Management]
    │   │       ├── Log Analytics Workspace
    │   │       ├── Automation Account
    │   │       └── Data Collection Rules
    │   │
    │   ├── connectivity（ネットワーク）─ ネットワーク基盤
    │   │   └── [Subscription: Connectivity]
    │   │       ├── Hub VNet（東日本・西日本）
    │   │       ├── Firewall
    │   │       ├── Bastion
    │   │       └── VPN/ExpressRoute Gateway
    │   │
    │   ├── identity（ID管理）─ 認証・認可システム
    │   │   └── [Subscription: Identity]
    │   │       └── Azure AD Domain Services等
    │   │
    │   └── security（セキュリティ）─ セキュリティ監視
    │       └── [Subscription: Security]
    │           └── Defender for Cloud等
    │
    ├── landingzones（ランディングゾーン層）─ アプリケーション
    │   ├── corp（社内アプリ）─ 社内システム用
    │   │   └── [Subscription: Corp-App]
    │   │       └── 会計システム、人事システム等
    │   │
    │   └── online（公開アプリ）─ インターネット公開用
    │       └── [Subscription: Online-App]
    │           └── Webサービス、API等
    │
    ├── sandbox（実験環境）─ 検証・テスト用
    │   └── [Subscription: Sandbox]
    │       └── 開発者が自由に使える
    │
    └── decommissioned（廃止予定）─ 削除前の一時保管
        └── [Subscription: Old-System]
            └── 削除予定のリソース

    🎯 各層の役割まとめ

    管理グループ 役割 誰が使う ポリシー
    platform インフラ基盤 インフラチーム 厳しい（変更制限）
    - management ログ・監視 インフラチーム 削除禁止
    - connectivity ネットワーク ネットワークチーム 変更要承認
    - identity 認証・認可 セキュリティチーム 最高レベル
    - security セキュリティ監視 セキュリティチーム 最高レベル
    landingzones アプリケーション 開発チーム 標準
    - corp 社内アプリ 開発チーム 標準+社内接続のみ
    - online 公開アプリ 開発チーム 標準+インターネット許可
    sandbox 実験環境 全員 緩い（自由に実験）
    decommissioned 廃止予定 削除担当者 読み取り専用 
    Tenant Root Group（テナントルート）
  └── alz（Azure Landing Zones）
      ├── platform（プラットフォーム）
      │   ├── management（管理）
      │   ├── connectivity（ネットワーク）
      │   ├── identity（ID管理）
      │   └── security（セキュリティ）
      ├── landingzones（ランディングゾーン）
      │   ├── corp（社内アプリ）
      │   └── online（インターネット公開）
      ├── sandbox（検証環境）
      └── decommissioned（廃止予定）

    階層のルール： - 最大6階層まで - 1つのサブスクリプションは1つの管理グループにのみ所属 - ポリシーは子に継承される

    なんで階層化するの？¶

    理由1：ガバナンス 

    platform配下：厳しいセキュリティポリシー
sandbox配下：緩いポリシー（実験用）

    理由2：権限管理 

    platformの管理者：platform配下だけ触れる
landingzonesの管理者：landingzones配下だけ触れる

    理由3：コスト管理 

    platform：インフラコスト
landingzones：アプリケーションコスト
→ 分けて集計できる

    Part 1: アーキテクチャ定義（alz_custom.alz_architecture_definition.yaml）¶

    ファイル構造¶

    lib/
  └── architecture_definitions/
      └── alz_custom.alz_architecture_definition.yaml  # ←これ

    内容を見てみよう¶

    name: alz_custom
management_groups:
  - id: alz
    display_name: Azure Landing Zones
    archetypes:
      - root_custom
    exists: false
    parent_id: null

    フィールド解説：

    id¶

    id: alz

    何？：管理グループのID（プログラム内で使う名前）

    ルール： - 小文字英数字とハイフンのみ - 一意（重複不可）

    display_name¶

    display_name: Azure Landing Zones

    何？：表示名（人間が見る名前）

    Azureポータルでこの名前が表示される。

    archetypes¶

    archetypes:
  - root_custom

    何？：適用するポリシーテンプレート

    lib/archetype_definitions/root_custom.alz_archetype_override.yamlを参照。

    exists¶

    exists: false

    何？：既に存在するか？

    • false：Terraformで作成
    • true：既存（参照のみ）

    parent_id¶

    parent_id: null

    何？：親管理グループのID

    • null：テナントルートの直下
    • alz：alzの配下

    階層の全体像¶

    # ルート
- id: alz
  parent_id: null  # ←テナントルート直下

# プラットフォーム
- id: platform
  parent_id: alz  # ←alzの配下

# 管理グループ（プラットフォーム配下）
- id: management
  parent_id: platform  # ←platformの配下

- id: connectivity
  parent_id: platform

- id: identity
  parent_id: platform

- id: security
  parent_id: platform

# ランディングゾーン
- id: landingzones
  parent_id: alz

- id: corp
  parent_id: landingzones  # ←landingzonesの配下

- id: online
  parent_id: landingzones

# その他
- id: sandbox
  parent_id: alz

- id: decommissioned
  parent_id: alz

    図にするとこう： 

    alz
├── platform
│   ├── management
│   ├── connectivity
│   ├── identity
│   └── security
├── landingzones
│   ├── corp
│   └── online
├── sandbox
└── decommissioned

    アーキタイプって何？¶

    **ポリシーのテンプレート**みたいなもん。

    例： - root_custom：ルート用のポリシーセット - platform_custom：プラットフォーム用 - landing_zones_custom：ランディングゾーン用

    ファイル構造¶

    lib/
  └── archetype_definitions/
      ├── root_custom.alz_archetype_override.yaml
      ├── platform_custom.alz_archetype_override.yaml
      ├── management_custom.alz_archetype_override.yaml
      ├── connectivity_custom.alz_archetype_override.yaml
      ├── identity_custom.alz_archetype_override.yaml
      ├── security_custom.alz_archetype_override.yaml
      ├── landing_zones_custom.alz_archetype_override.yaml
      ├── corp_custom.alz_archetype_override.yaml
      ├── online_custom.alz_archetype_override.yaml
      ├── sandbox_custom.alz_archetype_override.yaml

name: root_custom
policy_assignments_to_add: []
policy_assignments_to_remove: []

    フィールド解説：

    base_archetype¶

    base_archetype: root

    何？：ベースとなるアーキタイプ

    ALZモジュールに組み込まれてるrootアーキタイプを継承。

    policy_assignments_to_add¶

    policy_assignments_to_add: []

    何？：追加するポリシー割り当て

    例（カスタムポリシーを追加したい場合）： 

    policy_assignments_to_add:
  - name: custom-tagging-policy
    policy_definition_id: /providers/Microsoft.Management/managementGroups/alz/providers/Microsoft.Authorization/policyDefinitions/custom-tags
    parameters:
      requiredTags:
        value: ["environment", "owner"]

    policy_assignments_to_remove¶

    policy_assignments_to_remove: []

    何？：削除するポリシー割り当て

    例（デフォルトのポリシーを無効化したい場合）： 

    policy_assignments_to_remove:
  - Deploy-MDFC-DefenderSQL-AMA

    Chapter 3で見た設定です。

    architecture_name = "alz"

    何？：使用するアーキテクチャ定義の名前

    lib/architecture_definitions/alz_custom.alz_architecture_definition.yamlのnameフィールドと一致させる。

    location¶

    location = "{{starter_location_01}}"

    何？：管理グループのメタデータを保存する場所

    実は管理グループ自体は場所を持たないんだけど、ポリシー割り当てのデプロイメント用。

    policy_default_values¶

    policy_default_values = {
  log_analytics_workspace_id = "{{log_analytics_workspace_id}}"
  ...
}

    何？：ポリシーのデフォルトパラメータ

    多くのポリシーが「Log Analyticsにログ送信」を要求するから、ワークスペースIDをここで一元管理。

    メリット： 

    # これやらなくていい（自動で設定される）
policy_assignment "logging" {
  parameters = {
    log_analytics_workspace_id = "/subscriptions/.../workspaces/law-jp"
  }
}

    subscription_placement¶

    subscription_placement = {
  management   = ["{{subscription_id_management}}"]
  connectivity = ["{{subscription_id_connectivity}}"]
  identity     = ["{{subscription_id_identity}}"]
}

    何？：サブスクリプションをどの管理グループに配置するか

    例： 

    subscription_placement = {
  management   = ["12345678-1234-1234-1234-123456789012"]
  connectivity = ["87654321-4321-4321-4321-210987654321"]
  corp         = ["aaaaaaaa-bbbb-cccc-dddd-eeeeeeeeeeee", "ffffffff-gggg-hhhh-iiii-jjjjjjjjjjjj"]
}

    結果： 

    management グループ
  └── サブスクリプション 12345678-...

connectivity グループ
  └── サブスクリプション 87654321-...

corp グループ
  ├── サブスクリプション aaaaaaaa-...
  └── サブスクリプション ffffffff-...

    ポリシーの仕組み¶

    ポリシーの種類¶

    1. Audit（監査）¶

    「これやっちゃダメだけど、やってもエラーにはしない」
→ ポリシー違反として記録

    例： - パブリックIPを作成したら警告 - NSGがないVMを作成したら警告

    2. Deny（拒否）¶

    「これやっちゃダメ。やろうとしたらエラー」
→ リソース作成を拒否

    例： - 特定リージョン以外でのリソース作成を拒否 - 暗号化なしのストレージ作成を拒否

    3. DeployIfNotExists（自動デプロイ）¶

    「これがなかったら自動で作る」
→ 不足しているリソースを自動作成します

    例： - VMを作ったら自動でMonitoring Agent追加 - Diagnostic Settings自動設定

    4. Modify（変更）¶

    「タグとか設定を自動で変更」
→ リソースの属性を変更

    例： - タグを自動追加 - NSGルールを自動追加

    ポリシーの割り当て¶

    管理グループにポリシー割り当て
  ↓
子管理グループに継承
  ↓
サブスクリプションに継承
  ↓
リソースグループに適用
  ↓
リソースに適用

    例： 

    alz（ルート）
  └── ポリシー：「japaneast以外禁止」
      ↓
platform
  └── 継承：「japaneast以外禁止」
      ↓
connectivity
  └── 継承：「japaneast以外禁止」
      ↓
サブスクリプション
  └── 継承：「japaneast以外禁止」
      ↓
リソース作成時にチェック

    実践：カスタムポリシーを追加してみよう¶

    例1: タグ必須化¶

    1. アーキタイプファイル編集¶

    # lib/archetype_definitions/root_custom.alz_archetype_override.yaml
base_archetype: root
name: root_custom
policy_assignments_to_add:
  - name: Require-Tags
    display_name: "Require specific tags on resources"
    policy_definition_id: /providers/Microsoft.Authorization/policyDefinitions/96670d01-0a4d-4649-9c89-2d3abc0a5025
    parameters:
      tagNames:
        value:
          - environment
          - owner
          - cost_center

    2. Terraform実行¶

    terraform plan
# ↓ポリシー割り当てが追加される

terraform apply

    3. 結果¶

    リソース作成時：
❌ タグなし → エラー
❌ environment だけ → エラー（owner、cost_centerがない）
✅ 3つ全部ある → OK

    例2: 既存ポリシーの削除¶

    Chapter 3で見たDefender for SQLを無効化する例。

    # lib/archetype_definitions/root_custom.alz_archetype_override.yaml
policy_assignments_to_remove:
  - Deploy-MDFC-DefenderSQL-AMA

    理由： - コストがかかる - 使ってないから無効化したい

    デバッグ技¶

    管理グループの確認¶

    # 管理グループ一覧
az account management-group list --output table

# 特定の管理グループの詳細
az account management-group show --name alz

# 階層構造を表示
az account management-group show --name alz --expand --recurse

    ポリシー割り当ての確認¶

    # 管理グループのポリシー割り当て一覧
az policy assignment list --scope /providers/Microsoft.Management/managementGroups/alz

# 特定のポリシー割り当ての詳細
az policy assignment show --name "Require-Tags" --scope /providers/Microsoft.Management/managementGroups/alz

    ポリシーコンプライアンスの確認¶

    # コンプライアンス状態
az policy state list --resource /providers/Microsoft.Management/managementGroups/alz

↓
管理グループごとにコンプライアンス率が見える

    よくあるエラー¶

    エラー1: 管理グループ名の重複¶

    Error: Management group with id "alz" already exists

    原因：既に同じIDの管理グループが存在

    対処法： 

    # alz_custom.alz_architecture_definition.yaml
- id: alz
  exists: true  # ←既存使う

    または：

    # 既存削除
az account management-group delete --name alz

    エラー2: サブスクリプションの移動エラー¶

    Error: Cannot move subscription to management group

    原因：権限不足

    対処法： 

    # 必要な権限
# - Management Group Contributor
# - または Owner

# 権限付与
az role assignment create \
  --role "Management Group Contributor" \
  --assignee your@email.com \
  --scope /providers/Microsoft.Management/managementGroups/alz

    エラー3: ポリシー定義が見つからない¶

    Error: Policy definition not found

    原因：policy_definition_idが間違ってる

    対処法： 

    # ポリシー定義を検索
az policy definition list --query "[?displayName=='Require a tag'].id"

# 正しいIDをコピーして使う

    管理グループの設計パターン¶

    パターン1: 標準ALZ（推奨）¶

    alz
├── platform
│   ├── management
│   ├── connectivity
│   ├── identity
│   └── security
├── landingzones
│   ├── corp
│   └── online
└── sandbox

    メリット：Microsoftのベストプラクティス デメリット：複雑

    パターン2: シンプル構成¶

    company
├── production
├── development
└── sandbox

    メリット：わかりやすい デメリット：スケールしにくい

    パターン3: 環境 × 機能¶

    company
├── prod
│   ├── apps
│   ├── data
│   └── infrastructure
├── dev
│   ├── apps
│   ├── data
│   └── infrastructure
└── sandbox

    メリット：柔軟 デメリット：管理が複雑

    おすすめ：標準ALZ + カスタマイズ¶

    alz
├── platform（そのまま）
├── landingzones
│   ├── corp
│   ├── online
│   └── app-specific（アプリ専用追加）
└── sandbox

    標準構造を維持しつつ、必要に応じて追加。

    まとめ¶

    管理グループの役割： 1. 階層化：サブスクリプションをグルーピング 2. ガバナンス：ポリシーの一元管理 3. 権限管理：RBACの継承 4. コスト管理：階層ごとに集計

    ポリシーの役割： 1. Audit：監査（警告） 2. Deny：拒否（エラー） 3. DeployIfNotExists：自動デプロイ 4. Modify：自動変更

    覚えておくこと： - アーキテクチャ定義（YAML）で階層構造を定義 - アーキタイプ（YAML）でポリシーセットを定義 - ポリシーは親から子に継承される - policy_assignments_to_removeでデフォルトポリシーを無効化できる

    次のChapterでは、管理リソース（Log Analytics、DCR、Managed Identity）の作成を見ていきます。 監視とログ収集の基盤です。
