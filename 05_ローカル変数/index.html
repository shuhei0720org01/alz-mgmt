    05. ローカル変数 - locals.tfの役割¶

    このChapterでやること¶

    locals.tfの役割を理解する。

    例えるなら： - variables.tf：材料（卵、砂糖、小麦粉） - locals.tf：下ごしらえ（卵を溶く、材料を混ぜる） - main.tf：調理（オーブンで焼く）

    変数をそのまま使うんじゃなくて、計算・加工してから使う。

    locals（ローカル変数）とは？¶

    定義方法¶

    locals {
  project_name = "alz"
  full_name    = "${local.project_name}-${var.environment}"
}

    使い方： 

    resource "azurerm_resource_group" "example" {
  name     = local.full_name  # ←local.で参照
  location = var.location     # ←変数はvar.で参照
}

    ポイント： - 定義：locals { ... }（複数形） - 参照：local.xxx（単数形）

    variablesとの違い¶

    初心者がよく混乱するポイントです。しっかり理解しましょう。

    項目 variables.tf locals.tf
    外部から設定 ✅ できる ❌ できない
    計算・加工 ❌ できない ✅ できる
    参照方法 var.xxx local.xxx
    使う場面 ユーザーが設定する値 計算した値
    変更のタイミング 実行時に指定 コード内で固定

    具体例で理解する:

    # variables.tf（ユーザーが決める）
variable "environment" {
  type = string
  default = "prod"
}

variable "location" {
  type = string
  default = "japaneast"
}

# locals.tf（自動で計算する）
locals {
  # environmentとlocationを組み合わせてリソース名を作る
  resource_group_name = "rg-${var.environment}-${var.location}"
  # → "rg-prod-japaneast"

  # 複雑な条件判定
  is_production = var.environment == "prod" ? true : false

  # 計算結果
  subnet_count = var.enable_vpn ? 5 : 4
}

    どっちを使えばいいか迷ったら:

    ユーザーに変更させたい？
  YES → variables
  NO  → locals

    📊 処理フローの視覚化

    【Terraform実行の流れ】

1. 入力（variables）
   ↓
   var.environment = "prod"
   var.location = "japaneast"
   var.enable_vpn = true
   ↓

2. 加工（locals）
   ↓
   local.resource_group_name = "rg-prod-japaneast"  ← 文字列結合
   local.is_production = true                        ← 条件判定
   local.subnet_count = 5                            ← 計算
   ↓

3. 出力（resources）
   ↓
   resource "azurerm_resource_group" "example" {
     name     = local.resource_group_name  ← 加工済みの値を使う
     location = var.location               ← 入力値をそのまま使う
   }

    🎯 具体例でもう一度

    # ① ユーザーからの入力
variable "project" { default = "webapp" }
variable "env" { default = "prod" }
variable "region" { default = "japaneast" }

# ② locals で加工・計算
locals {
  # 名前を自動生成
  resource_group_name = "rg-${var.project}-${var.env}-${var.region}"
  # → "rg-webapp-prod-japaneast"

  # タグを自動生成
  common_tags = {
    Project     = var.project
    Environment = var.env
    ManagedBy   = "Terraform"
  }

  # 本番環境かどうか判定
  is_production = var.env == "prod"

  # 本番なら3台、それ以外は1台
  vm_count = local.is_production ? 3 : 1
}

# ③ resources で実際にリソース作成
resource "azurerm_resource_group" "main" {
  name     = local.resource_group_name  # ← 加工済み
  location = var.region                 # ← そのまま
  tags     = local.common_tags          # ← 加工済み
}

    覚え方: - variables: 入力（ユーザーが決める） - locals: 処理（自動で計算） - resources: 出力（実際に作る）

    Part 1: 定数定義¶

    locals {
  const = {
    connectivity = {
      virtual_wan        = "virtual_wan"
      hub_and_spoke_vnet = "hub_and_spoke_vnet"
      none               = "none"
    }
  }
}

    これは何？ 定数を定義しています。マジックストリング（文字列直書き）を避けるためです。

    悪い例： 

    if (var.connectivity_type == "hub_and_spoke_vnet") {  # ←タイポしやすい
  ...
}

    良い例： 

    if (var.connectivity_type == local.const.connectivity.hub_and_spoke_vnet) {
  # ←補完が効く、タイポしたらエラーになる
  ...
}

    定義されてる定数： - virtual_wan：Virtual WAN使う - hub_and_spoke_vnet：Hub-and-Spoke使う - none：ネットワーク作らない

    プログラミングのenumみたいなもん。

    Part 2: 接続タイプの判定¶

    locals {
  connectivity_enabled                    = var.connectivity_type != local.const.connectivity.none
  connectivity_virtual_wan_enabled        = var.connectivity_type == local.const.connectivity.virtual_wan
  connectivity_hub_and_spoke_vnet_enabled = var.connectivity_type == local.const.connectivity.hub_and_spoke_vnet
}

    これは何？ connectivity_typeの値を見て、フラグを作っています。

    connectivity_enabled¶

    connectivity_enabled = var.connectivity_type != local.const.connectivity.none

    意味：ネットワーク作るの？

    例： 

    # tfvars
connectivity_type = "hub_and_spoke_vnet"

# 結果
local.connectivity_enabled = true  # ←noneじゃないからtrue


    # tfvars
connectivity_type = "none"

# 結果
local.connectivity_enabled = false  # ←noneだからfalse

    使い道： 

    # ネットワーク作る時だけリソース作成
resource "azurerm_xxx" "example" {
  count = local.connectivity_enabled ? 1 : 0
  ...
}

    connectivity_virtual_wan_enabled¶

    connectivity_virtual_wan_enabled = var.connectivity_type == local.const.connectivity.virtual_wan

    意味：Virtual WAN使うの？

    例： 

    # tfvars
connectivity_type = "virtual_wan"

# 結果
local.connectivity_virtual_wan_enabled = true

    connectivity_hub_and_spoke_vnet_enabled¶

    connectivity_hub_and_spoke_vnet_enabled = var.connectivity_type == local.const.connectivity.hub_and_spoke_vnet

    意味：Hub-and-Spoke使うの？

    例： 

    # tfvars
connectivity_type = "hub_and_spoke_vnet"

# 結果
local.connectivity_hub_and_spoke_vnet_enabled = true

    なんでこんなことするの？

    main.tfで条件分岐しやすくするため。

    # これより
module "virtual_wan" {
  count = var.connectivity_type == "virtual_wan" ? 1 : 0
  ...
}

# こっちの方が読みやすい
module "virtual_wan" {
  count = local.connectivity_virtual_wan_enabled ? 1 : 0
  ...
}

    Part 3: リソースグループの依存関係¶

    locals {
  resource_groups = {
    resource_groups = module.resource_groups
  }
  hub_and_spoke_networks_settings = merge(module.config.outputs.hub_and_spoke_networks_settings, local.resource_groups)
  hub_virtual_networks            = (merge({ vnets = module.config.outputs.hub_virtual_networks }, local.resource_groups)).vnets
  virtual_wan_settings            = merge(module.config.outputs.virtual_wan_settings, local.resource_groups)
  virtual_hubs                    = (merge({ vhubs = module.config.outputs.virtual_hubs }, local.resource_groups)).vhubs
}

    これは何？ リソースグループを先に作ってから、ネットワークを作るための仕組み。

    Terraformの依存関係¶

    問題： 

    # リソースグループ
resource "azurerm_resource_group" "network" {
  name = "rg-network"
}

# VNet
resource "azurerm_virtual_network" "hub" {
  resource_group_name = azurerm_resource_group.network.name
  # ↑これで依存関係ができる（RG→VNet）
}

    でも、モジュール使ってる場合は**明示的に依存関係を書く**必要がある。

    resource_groups¶

    resource_groups = {
  resource_groups = module.resource_groups
}

    リソースグループモジュールの出力を保存。

    merge関数¶

    hub_and_spoke_networks_settings = merge(
  module.config.outputs.hub_and_spoke_networks_settings,
  local.resource_groups
)

    merge関数：2つのオブジェクトをマージする

    イメージ： 

    # 元のデータ
hub_and_spoke_networks_settings = {
  location = "japaneast"
  cidr     = "10.0.0.0/16"
}

# マージ後
hub_and_spoke_networks_settings = {
  location = "japaneast"
  cidr     = "10.0.0.0/16"
  resource_groups = module.resource_groups  # ←追加
}

    効果： - resource_groupsを含めることで、Terraformが自動で依存関係を理解する - リソースグループ → ネットワークの順番で作られる

    hub_virtual_networks¶

    hub_virtual_networks = (merge({ vnets = module.config.outputs.hub_virtual_networks }, local.resource_groups)).vnets

    **ちょっと複雑**だから分解しよう。

    ステップ1: mergeする¶

    merge(
  { vnets = module.config.outputs.hub_virtual_networks },
  local.resource_groups
)

# 結果
{
  vnets = { ... }
  resource_groups = module.resource_groups
}

    ステップ2: .vnetsで取り出す¶

    (...).vnets

# 結果
{ ... }  # ←vnetsの中身だけ

    なんでこんなことするの？ - resource_groupsを含めて依存関係を作る - でも最終的にはvnetsだけほしい

    トリッキーだけど、Terraformの依存関係システムを満たすための技。

    Part 4: ポリシーの依存関係¶

    locals {
  management_group_dependencies = {
    policy_assignments = [
      module.management_resources,
      module.hub_and_spoke_vnet,
      module.virtual_wan
    ]
    policy_role_assignments = [
      module.management_resources,
      module.hub_and_spoke_vnet,
      module.virtual_wan
    ]
  }
}

    これは何？ 管理グループにポリシーを割り当てる前に、**他のリソースを先に作る**ための仕組み。

    なんで必要なの？¶

    シナリオ： 1. Log Analyticsワークスペースを作る 2. ポリシーで「全リソースはLog Analyticsにログ送信」を設定

    問題： ポリシーを先に作ると、Log Analyticsがまだないからエラーになる。

    解決策： 

    policy_assignments = [
  module.management_resources,  # ←これらを先に作る
  module.hub_and_spoke_vnet,
  module.virtual_wan
]

    リストに含めることで、Terraformが順番を理解する。

    policy_assignments vs policy_role_assignments¶

    policy_assignments： ポリシー定義を割り当てる（「ログ送信しなさい」）

    policy_role_assignments： ポリシーのマネージドIDに権限を与える（「ログ送信する権限あげるよ」）

    どっちも先に他のリソースを作る必要がある。

    Part 5: 最終的な設定のマージ¶

    management_group_settings¶

    locals {
  management_group_settings = merge(
    module.config.outputs.management_group_settings,
    {
      dependencies = local.management_group_dependencies
    }
  )
}

    やってること： 1. module.configから管理グループの設定を取得 2. dependencies（依存関係）を追加 3. マージして最終設定を作る

    イメージ： 

    # 元の設定
management_group_settings = {
  root_name = "alz"
  policies  = [ ... ]
}

# マージ後
management_group_settings = {
  root_name    = "alz"
  policies     = [ ... ]
  dependencies = {  # ←追加
    policy_assignments = [ ... ]
    policy_role_assignments = [ ... ]
  }
}

    management_resource_settings¶

    locals {
  management_resource_settings = merge(
    module.config.outputs.management_resource_settings,
    {
      tags = coalesce(module.config.outputs.management_resource_settings.tags, module.config.outputs.tags)
    }
  )
}

    **coalesce関数**って何？

    coalesce(値1, 値2, 値3, ...)
# ←最初のnullじゃない値を返す

    例： 

    coalesce(null, null, "hello")  # ←"hello"
coalesce("first", "second")    # ←"first"

    この場合： 

    tags = coalesce(
  module.config.outputs.management_resource_settings.tags,
  module.config.outputs.tags
)

    意味： - 管理リソース専用のタグがあればそれを使う - なければ全体のタグを使う

    使い道： 

    # 全体のタグ
tags = {
  environment = "prod"
}

# 管理リソース専用のタグ（指定してない）
management_resource_settings.tags = null

# 結果
# ↓全体のタグが使われる
management_resource_settings.tags = { environment = "prod" }

    locals.tfのパターンまとめ¶

    パターン1: 定数定義¶

    locals {
  const = {
    status = {
      active   = "active"
      inactive = "inactive"
    }
  }
}

# 使い方
status = local.const.status.active

    メリット：タイポ防止、補完が効く

    パターン2: 条件分岐¶

    locals {
  is_production = var.environment == "prod"
  instance_count = local.is_production ? 3 : 1
}

    メリット：複雑な条件をわかりやすく

    パターン3: データ加工¶

    locals {
  location_short = {
    "japaneast" = "jpe"
    "japanwest" = "jpw"
  }

  resource_name = "${var.prefix}-${local.location_short[var.location]}"
}

    メリット：変換ロジックを1箇所に集約

    パターン4: 依存関係の明示¶

    locals {
  config_with_deps = merge(
    var.config,
    { dependencies = [module.base_resources] }
  )
}

    メリット：リソースの作成順序を制御

    パターン5: デフォルト値のフォールバック¶

    locals {
  tags = coalesce(var.custom_tags, var.default_tags)
}

    メリット：柔軟な設定、省略時の挙動を制御

    実践：locals.tfを書いてみよう¶

    例1: 環境別の設定¶

    # variables.tf
variable "environment" {
  type = string
}

# locals.tf
locals {
  env_config = {
    dev = {
      instance_size = "Standard_B2s"
      instance_count = 1
    }
    prod = {
      instance_size = "Standard_D4s_v3"
      instance_count = 3
    }
  }

  config = local.env_config[var.environment]
}

# main.tf
resource "azurerm_virtual_machine" "example" {
  count = local.config.instance_count
  size  = local.config.instance_size
  ...
}

    例2: 命名規則の統一¶

    # variables.tf
variable "project" { type = string }
variable "environment" { type = string }
variable "location" { type = string }

# locals.tf
locals {
  location_codes = {
    "japaneast" = "jpe"
    "japanwest" = "jpw"
  }

  naming_prefix = "${var.project}-${var.environment}-${local.location_codes[var.location]}"

  resource_names = {
    vnet          = "${local.naming_prefix}-vnet"
    subnet        = "${local.naming_prefix}-subnet"
    nsg           = "${local.naming_prefix}-nsg"
    resource_group = "rg-${local.naming_prefix}"
  }
}

# main.tf
resource "azurerm_resource_group" "example" {
  name = local.resource_names.resource_group
}

resource "azurerm_virtual_network" "example" {
  name = local.resource_names.vnet
}

    例3: 条件付きリソース作成¶

    # variables.tf
variable "enable_monitoring" { type = bool }
variable "enable_backup" { type = bool }

# locals.tf
locals {
  monitoring_enabled = var.enable_monitoring
  backup_enabled     = var.enable_backup

  # 両方有効な場合のみ高度な監視を有効化
  advanced_monitoring_enabled = local.monitoring_enabled && local.backup_enabled
}

# main.tf
resource "azurerm_log_analytics_workspace" "example" {
  count = local.monitoring_enabled ? 1 : 0
  ...
}

resource "azurerm_recovery_services_vault" "example" {
  count = local.backup_enabled ? 1 : 0
  ...
}

resource "azurerm_monitor_diagnostic_setting" "advanced" {
  count = local.advanced_monitoring_enabled ? 1 : 0
  ...
}

    デバッグ技：locals の値を確認する¶

    output使おう

    # outputs.tf
output "debug_locals" {
  value = {
    connectivity_enabled = local.connectivity_enabled
    is_production        = local.is_production
    resource_names       = local.resource_names
  }
}

    実行： 

    terraform plan

# 出力
Outputs:
  debug_locals = {
    connectivity_enabled = true
    is_production = false
    resource_names = {
      vnet = "myapp-dev-jpe-vnet"
      ...
    }
  }

    ローカル変数の中身が見えるから、デバッグが楽。

    まとめ¶

    locals.tfの役割： 1. 定数定義：マジックストリングを避ける 2. 条件判定：フラグを作って可読性向上 3. 依存関係：リソースの作成順序を制御 4. データ加工：変数を計算・変換 5. デフォルト値：フォールバック処理

    覚えておくこと： - 定義：locals { ... }（複数形） - 参照：local.xxx（単数形） - 外部から値を渡せない（内部計算専用） - merge()、coalesce()を使いこなそう

    練習問題¶

    理解度チェックです。

    問題1¶

    以下のコードで、local.resource_group_nameの値は何になりますか？

    variable "environment" {
  default = "prod"
}

variable "location" {
  default = "japaneast"
}

locals {
  resource_group_name = "rg-${var.environment}-${var.location}"
}

    問題2¶

    variablesとlocalsの違いを説明してください。
    どんな時にlocalsを使うべきですか？

    問題3¶

    以下のコードは何をしていますか？

    locals {
  is_production = var.environment == "prod" ? true : false
}

    練習問題の答え¶

    答え1¶

    rg-prod-japaneast

    文字列補間でvar.environment（prod）とvar.location（japaneast）が結合されます。

    答え2¶

    違い: - variables: 外部から値を受け取る（ユーザーが設定） - locals: 内部で計算する（自動で決まる）

    localsを使うべき時: - 複数の変数を組み合わせて新しい値を作る時 - 条件判定をして値を決める時 - 同じ計算を繰り返し使う時（DRY原則）

    答え3¶

    **三項演算子**を使って、環境が本番（prod）かどうかを判定しています。

    var.environment == "prod" ? true : false
         ↑                  ↑      ↑
      条件式             真の場合 偽の場合
    • environmentが"prod"ならtrue
    • それ以外ならfalse

    この値を使って、本番環境だけ特別な設定を適用できます。
