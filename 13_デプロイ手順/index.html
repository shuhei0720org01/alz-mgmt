    Step 1-2: Service Principal作成（OIDC用）¶

    🔐 OIDC認証とは？（初心者向け説明）

    従来の方法（パスワード認証）: 

    GitHub → Azure
「パスワードは "abc123" です」

    問題点: - ❌ パスワードが漏れたら危険 - ❌ 定期的な更新が必要 - ❌ GitHubに保存するのが不安

    新しい方法（OIDC認証）: 

    GitHub → Azure
「私はGitHubの公式リポジトリですよ（証明書提示）」
Azure → GitHub
「本当だ。じゃあ信じる」

    メリット: - ✅ パスワード不要（証明書ベース） - ✅ 自動更新 - ✅ 漏洩リスクが低い

    Federated Credential（フェデレーテッド資格情報）とは？: 「このGitHubリポジトリを信頼する」という設定。 特定のGitHubリポジトリからのリクエストだけを許可します。

    Federated Credential = 「信頼するリポジトリのリスト」

    具体例: 

    repo:shuhei0720org01/alz-mgmt:ref:refs/heads/main
 ↑         ↑             ↑          ↑
 |         |             |          mainブランチ
 |         |             リポジトリ
 |         オーナー
 プラットフォーム(GitHub)

    これで、shuhei0720org01/alz-mgmt の main ブランチからだけアクセスできるようになります。

    では、実際に設定していきましょう:

    # 変数設定
APP_NAME="github-actions-alz-mgmt"
REPO_OWNER="your-github-username"
REPO_NAME="alz-mgmt"
SUBSCRIPTION_ID=$(az account show --query id -o tsv)

# Azure ADアプリ作成
az ad app create --display-name "$APP_NAME"

# App IDを取得
APP_ID=$(az ad app list --display-name "$APP_NAME" --query "[0].appId" -o tsv)
echo "APP_ID: $APP_ID"

# Service Principal作成
az ad sp create --id $APP_ID

# Owner権限付与（Subscription全体）
az role assignment create \
  --assignee $APP_ID \
  --role "Owner" \
  --scope "/subscriptions/$SUBSCRIPTION_ID"

# Federated Credential作成（mainブランチ用）
az ad app federated-credential create \
  --id $APP_ID \
  --parameters "{
    \"name\": \"github-actions-main\",
    \"issuer\": \"https://token.actions.githubusercontent.com\",
    \"subject\": \"repo:${REPO_OWNER}/${REPO_NAME}:ref:refs/heads/main\",
    \"audiences\": [\"api://AzureADTokenExchange\"]
  }"

# Federated Credential作成（PR用）
az ad app federated-credential create \
  --id $APP_ID \
  --parameters "{
    \"name\": \"github-actions-pr\",
    \"issuer\": \"https://token.actions.githubusercontent.com\",
    \"subject\": \"repo:${REPO_OWNER}/${REPO_NAME}:pull_request\",
    \"audiences\": [\"api://AzureADTokenExchange\"]
  }"

    保存しておく値： 

    echo "AZURE_CLIENT_ID: $APP_ID"
echo "AZURE_TENANT_ID: $(az account show --query tenantId -o tsv)"
echo "AZURE_SUBSCRIPTION_ID: $SUBSCRIPTION_ID"

    メモっておこう！ あとで使う。

    Step 1-3: Storage Account作成（Terraform State用）¶

    # 変数設定
LOCATION="japaneast"
RG_NAME="rg-terraform-state"
STORAGE_ACCOUNT_NAME="stterraform$(openssl rand -hex 4)"  # ランダムな名前
CONTAINER_NAME="tfstate"

# Resource Group作成
az group create \
  --name $RG_NAME \
  --location $LOCATION

# Storage Account作成
az storage account create \
  --name $STORAGE_ACCOUNT_NAME \
  --resource-group $RG_NAME \
  --location $LOCATION \
  --sku Standard_LRS \
  --kind StorageV2 \
  --allow-blob-public-access false \
  --min-tls-version TLS1_2

# コンテナ作成
az storage container create \
  --name $CONTAINER_NAME \
  --account-name $STORAGE_ACCOUNT_NAME \
  --auth-mode login

# Service Principalに権限付与
STORAGE_ACCOUNT_ID=$(az storage account show \
  --name $STORAGE_ACCOUNT_NAME \
  --resource-group $RG_NAME \
  --query id -o tsv)

az role assignment create \
  --assignee $APP_ID \
  --role "Storage Blob Data Contributor" \
  --scope $STORAGE_ACCOUNT_ID

    保存しておく値： 

    echo "ARM_STORAGE_ACCOUNT_NAME: $STORAGE_ACCOUNT_NAME"
echo "ARM_CONTAINER_NAME: $CONTAINER_NAME"
echo "ARM_KEY: alz-mgmt.tfstate"
echo "ARM_RESOURCE_GROUP_NAME: $RG_NAME"

    ### Step 2-3: 設定ファイル編集

**platform-landing-zone.auto.tfvars** を自分の環境に合わせて編集：

```hcl
# 最小限の設定例
starter_location_01 = "japaneast"
starter_location_02 = "japanwest"

root_id                     = "myorg"
root_name                   = "My Organization"
default_location            = "japaneast"
default_postfix             = "001"
subscription_id_management  = "<YOUR_MANAGEMENT_SUBSCRIPTION_ID>"
subscription_id_identity    = "<YOUR_IDENTITY_SUBSCRIPTION_ID>"
subscription_id_connectivity = "<YOUR_CONNECTIVITY_SUBSCRIPTION_ID>"

# 最小構成（コスト削減）
connectivity_type = "hub_and_spoke_vnet"

hub_and_spoke_networks_settings = {
  enabled_resources = {
    ddos_protection_plan = false  # ←無効化
  }
}

hub_virtual_networks = {
  primary = {
    location = "japaneast"

    enabled_resources = {
      firewall                              = false  # ←無効化
      bastion                               = false  # ←無効化

    Phase 3: ローカルでの初回デプロイ¶

    なぜローカルで？ 

    GitHub Actionsでいきなりやると：
- エラー時のデバッグが大変
- ログ見るのが面倒

ローカルで：
- すぐエラーわかる
- 設定ファイル調整しやすい

    Step 3-1: Terraform初期化¶

    # 環境変数設定
export ARM_STORAGE_ACCOUNT_NAME="<Step 1-3の値>"
export ARM_CONTAINER_NAME="tfstate"
export ARM_KEY="alz-mgmt.tfstate"
export ARM_RESOURCE_GROUP_NAME="rg-terraform-state"

# Azureログイン（Owner権限のアカウント）
az login

# 使うSubscriptionを選択
az account set --subscription "<SUBSCRIPTION_ID>"

# Terraform初期化
terraform init

# Backend設定の確認
terraform init \
  -backend-config="storage_account_name=$ARM_STORAGE_ACCOUNT_NAME" \
  -backend-config="container_name=$ARM_CONTAINER_NAME" \
  -backend-config="key=$ARM_KEY" \
  -backend-config="resource_group_name=$ARM_RESOURCE_GROUP_NAME"

    成功すると： 

    Terraform has been successfully initialized!

    Step 3-2: Terraform Validate¶

    terraform validate

    Step 5-1: 設定ファイルを本番仕様に¶

    git checkout -b feature/enable-production-resources

    platform-landing-zone.auto.tfvars 編集：

    # Firewall有効化
hub_virtual_networks = {
  primary = {
    location = "japaneast"

    enabled_resources = {
      firewall                              = true   # ←有効化
      bastion                               = true   # ←有効化
      virtual_network_gateway_express_route = false
      virtual_network_gateway_vpn           = false  # ←必要なら有効化
      private_dns_zones                     = true
      private_dns_resolver                  = false
    }

    hub_virtual_network = {
      name          = "vnet-myorg-jpe-hub"
      address_space = ["10.0.0.0/16"]
      routing_address_space         = ["10.0.0.0/22"]
      route_table_name_firewall     = "rt-myorg-jpe-firewall"
      route_table_name_user_subnets = "rt-myorg-jpe-user"
    }

    firewall = {
      subnet_address_prefix            = "10.0.0.0/26"
      management_subnet_address_prefix = "10.0.0.64/26"
      name                             = "fw-myorg-jpe-hub"

      default_ip_configuration = {
        public_ip_config = {
          name = "pip-fw-myorg-jpe-hub"
        }
      }

      management_ip_enabled = true
      management_ip_configuration = {
        public_ip_config = {
          name = "pip-fw-myorg-jpe-hub-mgmt"
        }
      }
    }

    練習問題¶

    実践的な理解度チェックです。

    問題1¶

    OIDC認証とパスワード認証の違いを3つ挙げてください。

    問題2¶

    以下のエラーが出た場合、何が原因ですか？ 

    Error: zones are not supported in this region

    問題3¶

    Terraform Stateファイルをローカルではなく、
    Azure Storage Accountに保存する理由を2つ挙げてください。

    練習問題の答え¶

    答え1¶

    OIDC認証とパスワード認証の違い:

    1. セキュリティ: OIDC = パスワード不要、パスワード認証 = 漏洩リスクあり
    2. 更新: OIDC = 自動更新、パスワード認証 = 定期的な手動更新が必要
    3. 信頼モデル: OIDC = 証明書ベース、パスワード認証 = 秘密情報の保存

    答え2¶

    原因: Japan East等のリージョンで、Availability Zonesに対応していないリソースに対して
    zones = ["1", "2", "3"]のように指定している。

    解決策: zones = []（空リスト）を指定する。

    primary_firewall_client_public_ip_zones = []

    答え3¶

    Terraform Stateをリモート保存する理由:

    1. チーム共有: 複数人で同じStateを参照できる。ローカルだと個人のPCにしか保存されない。
    2. 安全性: ローカルPCの故障やファイル削除でStateが失われるリスクを回避。Azure Storageはバックアップされている。

    追加の利点: - State Lockingでチーム内の競合を防げる - GitHub Actionsから直接アクセスできる
